# GitHub и файлы

Этот документ описывает, какие файлы можно загружать в публичный репозиторий, а какие нельзя.

## Что можно загружать

Обычно можно загружать:

- исходный код проекта;
- `package.json`;
- `package-lock.json`;
- `requirements-fox.txt`;
- `vercel.json`;
- markdown-документацию;
- конфигурацию без секретов.

## Что нельзя загружать в публичный репозиторий

Не загружайте:

- `.env`;
- `.env.local`;
- `.env.production`;
- любые другие `.env*` файлы с реальными значениями;
- `node_modules/`;
- локальные профили браузера;
- локальные базы данных с приватной информацией;
- `backend/data/app.db`;
- `backend/data/app.db-journal`;
- `backend/data/app.db-wal`;
- `backend/data/app.db-shm`;
- временные debug-файлы;
- `playwright-debug/`;
- `screenshots/`;
- любые секреты и токены;
- логины, пароли, API-ключи;
- proxy credentials;
- дампы баз данных с приватными данными.

## Почему нельзя загружать `.env`

Файл `.env` часто содержит:

- токены;
- пароли;
- адреса приватных API;
- настройки прокси;
- ключи сервисов.

Если такой файл попадет в публичный GitHub, секреты нужно считать скомпрометированными и сразу заменить.

## Как безопасно описывать ENV

Можно создать пример без реальных значений:

```bash
PORT=3000
VITE_API_URL=https://example.com
```

Нельзя писать реальные секреты:

```bash
PLAYWRIGHT_PROXY_PASSWORD=real-password-here
TOKEN=real-token-here
```

## Проверка перед commit

Перед commit проверьте список файлов:

```bash
git status
```

Проверьте, что среди файлов нет:

- `.env`;
- `node_modules`;
- локальных баз данных;
- screenshots;
- debug-файлов;
- секретов и токенов.

## Если секрет уже попал в GitHub

1. Сразу замените секрет в сервисе, где он был создан.
2. Удалите секрет из репозитория.
3. Проверьте историю Git, потому что простое удаление файла не всегда удаляет секрет из истории.
4. При необходимости пересоздайте ключи, токены и пароли.
